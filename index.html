GitHub Insights App

Introduction:

  • A Github Insight App is designed to assist developers with daily routine tasks share useful insights from GitHub associated with developer productivity and enforce consistent development process implementation across all development.

  • It assists developers in identifying and addressing adverse practices occurring within GitHub repositories by leveraging data-driven actions.

  • It offers an aesthetically pleasing terminal output to present repository-related data, facilitating the extraction of meaningful insights.

Getting Started:

  • Download the installer file
    • Windows: gia_installer_windows.ps1
    • Linux and MacOs: gia_installer_linux_or_Mac.sh
  • Run the installer file via below steps:
    • open the windows terminal(not command prompt, as it is unable to run the ps1 file), shell, mac Os shell respective to the operating system.
    • Navigate to the folder where you download the installer file by cd \<path> command. By default it will be download in user/download folder.
    • Run the file by typing the file name.
    • Downloading will be started now.
  • Verify the app is installed perfectly or not by running gia command.
  • If it prints description of app then you are good to go. Ensure to run gia configure command first.

2. gia configure:

  • It is designed for configuring the username, access token, and organization name required to execute commands specific to that particular configuration.

  • It will store the user-related configuration information which is asked by the configure command into the config file in the .gia folder, the config file is already configured at the install time.

  • Usage: gia configure

  • It will show the prompt that asks for the necessary configuration data from the user. Such as

    • Username: This refers to the GitHub username, which is used to retrieve information associated with that username.

    • Access-Token: This is the access token provided to users, granting them access to their GitHub data.

    • Organization Name: This field is required to specify the organization to run the commands within a the organization on GitHub.

3. gia list-stale-branches

  • This command is used to fetch all of the stale-branches for the specific repository.

  • Stale branches: Refer to those branches that have been successfully merged but have not been deleted after 20 days(default interval) from the merged time, you can specify the time-interval as well.

  • The command will read the organization value from the config file by default if the value is not provided by the -o option. The command is repo-spefic for -r <repo-value> and -o <org-value> -r <repo-value> options only.

  • Options:

    -o <value>, --organization=<value>, â€“organization <value>

    In value, you have to specify the value of the organization in which you have to fetch the stale branches.

    -r <value>, --repository=<value>, --repository <value>

    In value you have to specify the value of the repository name in which you want to fetch the stale branches.

    --filter <value>

    To filter the repositories by <value>, It will filter and run the command on those repositories in which the <value> is present anywhere in repository name. This filter option is not useful when run the command on specific repository and effective on only run the command on organization. i.e. This option is only effective when user calls the command without -o and -r both and when user calls command with -o option only.

  • The user is forced to enter the value of the chosen option. If the user wants to go for any of the option(s) and forgets or omits to enter the appropriate value then the command will give the error.

    flag: -e, --export

    It is the flag, if passed then it will store the output of the command in the csv format inside the .gia/output folder.

  • Scenarios:

    • If the user calls the command gia list-stale-branches then It will fetch the organization value from the config file and the stale branches of all repositories of organization will be shown. If you want to run the command for other organization you are a part of then try -o option.

    • If the user calls the command as gia list-stale-branches -o <orgval> -r <reponame> then the command will list the stale branches of the specified reponame of the specified orgval.

    • If the user calls the command as gia list-stale-branches -o <orgval> then the command will execute for all the repositories of organization specified in orgval by the user in the command. This command is designed to run the default command but for other organization.

    • If the user calls the command as gia list-stale-branches -r <reponame> then the command will execute for the repository specified in the reponame by the user in the command and the organization name will be read from the config file. The repository must be the part of the organization which is configured in the configure command otherwise it may lead to an error.

    Note: â€“organization=<orgval> and â€“organization <orgval> can be used instead of -o <orgval> and the same for -r as well.

6. gia release-branches-prepare

  • The command is designed to prepare the release branches for the repository. It will firstly create the release branch from the production if not exist and then pull request is created from the staging branch to the release branch this will only happen if staging branch and production branch has difference otherwise it will not create the release branch and will not prepare the pull request as well. (production -> (create branch) -> release -> (create PR from) -> staging -> release).If the release branch is already there then it will skip the release branch creation step. If PR is already exist then it will skip PR creation step and existing PR will not be shown to cli output.

  • Filter option is mendatory in this command.

  • The command will read the organization name from the config file by default if not provided in the command. The command is repo-spefic for -r <repo-value> and -o <org-value> -r <repo-value> options only.

  • Options:

    -o \<value>, --organization=<value>, â€“organization <value>

    In value you have to specify the value of the organization in which you have to prepare release branches.

    -r \<value>, --repository=<value>, --repository <value>

    In value you have to specify the value of the repository name in which you want to prepare release branches.

    --filter <value>
    Note: This option is required to run this command on organzation or all repositories of user-personal-account. i.e. when run command with -o options or without any options

    To filter the repositories by <value>, It will filter and run the command on those repositories in which the <value> is present anywhere in repository name. This filter option is not useful when run the command on specific repository and effective on only run the command on organization. i.e. This option is only effective when user calls the command without -o and -r both and when user calls command with -o option only.

  • The user is forced to enter the value of the chosen option. If the user wants to go for any of the option(s) and forgets or omits to enter the appropriate value then the command will give the error.

    flag: -e, --export

    It is the flag, if passed then it will store the output of the command in the csv format inside the .gia/output folder.

  • Scenarios:

    • If the user calls the command as gia release-branches-prepare then It will fetch the organization name from the config file and will create the release branch and PR from staging to release for all the repositories of the organization fetched from the config file. If you want to run the command for other organization you are a part of then try -o option.

    • If the user calls the command as gia release-branches-prepare -o <orgval> -r <reponame> then the command will give the result for the specified orgval and reponame.

    • If the user calls the command as gia release-branches-prepare -o <orgval> then the command will execute for the organization specified in orgval by the user in the command.

If release branch is not found in any repository, any action will not performed for this repository. If the PR is already created then the command will skip the PR creation step and the PR will not be shown to the user in cli output.
